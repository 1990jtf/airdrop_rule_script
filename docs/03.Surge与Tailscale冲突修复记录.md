# Surge 与 Tailscale 冲突修复记录

> 记录日期：2026-04-16
> 环境：Mac Studio + macOS + Surge + Tailscale

## 一、问题现象

在 Mac Studio 上同时运行 Surge 和 Tailscale 时出现以下情况：

- **Tailscale 开启**：无法访问 YouTube、Google 等需要代理的网站
- **Tailscale 关闭**：一切正常
- **网络环境**：Wi-Fi 连接

### Surge Dashboard 观察到的现象

- 分流正确，YouTube 流量确实走了代理节点（日本 BWG Snell）
- 连接状态显示"活跃"，也连上了代理服务器（45.78.63.164）
- **但只有上传（2KB），没有下载（0B）**
- 连接一直挂着（时长持续增加），始终无数据返回

## 二、根因分析

### 关键线索：路由表对比

通过 `netstat -rn -f inet | head -30` 对比 Tailscale 开启前后的路由表：

**Tailscale 关闭时：**
```
Destination        Gateway            Flags               Netif
default            link#38            UCSg                utun5    ← Surge 虚拟网卡
default            192.168.8.1        UGScIg              en1      ← 物理网卡
```

**Tailscale 开启时：**
```
Destination        Gateway            Flags               Netif
default            link#38            UCSg                utun5    ← Surge 虚拟网卡
default            192.168.8.1        UGScIg              en1      ← 物理网卡
default            link#40            UCSIg               utun6    ← Tailscale 新增！
```

### 问题本质

Tailscale 启动后向路由表**插入了一条指向 `utun6`（Tailscale 虚拟网卡）的 default 路由**。

这导致了三方冲突：

1. Surge 通过 `utun5` 接管流量
2. Surge 处理后需要把代理流量以"真实网络请求"身份发往代理服务器
3. 但出站流量被 Tailscale 的 `utun6` default 路由截走，进入 Tailscale 隧道被黑洞

这就解释了为什么：
- TCP 握手能成功（2KB 上传发出去了）
- 但返回数据无法回来（0B 下载）

## 三、排查步骤

### 1. 在 Surge Dashboard 确认分流

打开 Surge → Dashboard → Requests，观察目标流量：
- 策略是否正确走到代理
- 上传/下载字节数是否对称
- 远端地址是否为代理服务器 IP

**关键判断**：如果分流正确但下载为 0，基本就是路由层被劫持。

### 2. 对比路由表

```bash
# 分别在 Tailscale 关闭和开启状态下运行
netstat -rn -f inet | head -30
```

重点看 `default` 行，是否多出指向 `utun*` 的新路由。

### 3. 验证特定 IP 的路由路径

```bash
# 代理服务器 IP 是否被 Tailscale 劫持
route get 45.78.63.164
```

查看 `interface` 字段，应该是 `utun5`（Surge）或 `en1`（物理），**不应该是 Tailscale 的 utun**。

### 4. 检查 Tailscale 状态

```bash
tailscale status
tailscale netcheck
```

## 四、解决方案

### 核心操作：关闭 Tailscale 的 "Accept Routes"

**方法一：GUI 操作（推荐）**

点击 Mac 菜单栏 Tailscale 图标 → Preferences / 设置：

- ✅ 关闭 **"Use Tailscale subnets"** / **"Accept routes"**
- ✅ 关闭 **"Use Tailscale DNS settings"**（可选，避免 DNS 冲突）
- ✅ **Exit Node** 设为 **None**

**方法二：命令行操作**

```bash
# 关闭 accept-routes
sudo tailscale set --accept-routes=false

# 确保没有设置 exit-node
sudo tailscale set --exit-node=
```

### 修复后验证

```bash
# 1. 确认路由表正常
netstat -rn -f inet | head -5
# 期望：default 只有 utun5 (Surge) 和 en1 (物理) 两条，没有 utun6

# 2. 确认代理服务器 IP 路由正确
route get 45.78.63.164
# 期望：interface 是 utun5 (被 Surge 接管)
```

## 五、原因溯源

Tailscale 的 `--accept-routes` 默认在 macOS 客户端是**关闭**的，出现这条 default 路由的常见原因：

1. **曾手动开启过 accept-routes**：之后一直保留该设置
2. **Tailnet 内有 Subnet router 广告了 0.0.0.0/0**：当客户端 accept-routes 开启，就会接受这条全局路由
3. **设置了 Exit Node**：所有流量强制走指定节点

## 六、经验总结

### 同时使用 Surge 和 Tailscale 的配置要点

1. **Tailscale 不要接受全局路由**：`accept-routes` 保持关闭，或只接受特定子网
2. **Tailscale 不要设置 Exit Node**：会与 Surge 的代理分流直接冲突
3. **DNS 解析**：让 Tailscale 的 `*.ts.net` 和 `*.tailscale.net` 走系统 DNS 或 Tailscale 的 MagicDNS（100.100.100.100），不要被 Surge 劫持
4. **Tailscale 内部 IP 段直连**：
   - IPv4: `100.64.0.0/10`
   - IPv6: `fd7a:115c:a1e0::/48`

### Surge 配置中已保留的 Tailscale 兼容配置

```ini
[General]
skip-proxy = ..., 100.64.0.0/10, ...

[Rule]
# Tailscale 进程直连
PROCESS-NAME,Tailscale,DIRECT
PROCESS-NAME,tailscaled,DIRECT

# Tailscale 域名直连
DOMAIN-SUFFIX,tailscale.com,DIRECT
DOMAIN-SUFFIX,ts.net,DIRECT

# Tailscale 网段直连
IP-CIDR6,fd7a:115c:a1e0::/48,DIRECT,no-resolve
IP-CIDR,100.64.0.0/10,DIRECT,no-resolve

[Host]
*.tailscale.net = server:system
*.ts.net = server:system
```

## 七、排查思路提炼

当出现"代理连接建立但无数据返回"的情况时，按这个顺序排查：

1. **分流是否正确** → 看 Surge Dashboard 的策略列
2. **是否真的连上代理服务器** → 看远端地址
3. **上传/下载是否对称** → 如果只有上传没下载，是数据回路问题
4. **系统路由表** → `netstat -rn` 看有没有第三方 VPN/隧道插入 default 路由
5. **特定目标 IP 的路由** → `route get <IP>` 看最终从哪个接口出去

路由层的问题比分流规则的问题更底层，也更隐蔽，优先排查。
